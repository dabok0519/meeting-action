# 회의록 액션아이템 관리 서비스

회의록 텍스트를 붙여넣으면 AI가 **결정사항 / 액션아이템 / 논의사항**으로 구조화하고, 그 결과를 사용자가 검토·수정한 뒤 회의별로 조회·관리하는 웹서비스.

- **GitHub:** https://github.com/dabok0519/meeting-action
- **배포 URL:** http://100.26.29.22/
- **스택:** React(Vite) · FastAPI · SQLite · OpenRouter · AWS EC2(Ubuntu) + Nginx
    - LLM 모델은 `.env`의 `OPENROUTER_MODEL` 값 하나로 교체된다. **무료 티어가 목표였으나 현재는 임시로 저가 유료 모델을 쓴다** — 이유는 1번·4번에 솔직히 적었습니다.

## 1. 무엇을 만들었고 어디까지 됐는가

**필수 기능 1~6을 로컬부터 AWS 배포까지 끝까지 관통시켰다.** 남이 URL로 접속해 입력→분석→저장→조회→수정→삭제 전 과정을 쓸 수 있는 상태다. 그 위에 권장 기능 2개(서버사이드 필터·정렬, 완료체크·누적 대시보드)까지 구현했다.

### 된 것 — 필수 1~6

| # | 기능 | 상태 |
| --- | --- | --- |
| 1 | 관계형 데이터 모델 (회의록 1 : 액션아이템 N) | ✅ |
| 2 | AI 구조화 (JSON 형식 강제 + hallucination 방어) | ✅ |
| 3 | AI 호출 비동기 처리 (로딩·타임아웃·재시도) | ✅ |
| 4 | CRUD + 사용자 검토·수정 후 저장 | ✅ |
| 5 | 프론트(React) ↔ 백엔드(REST API) 분리 + 실제 HTTP + DB 저장 | ✅ |
| 6 | AWS EC2 배포 (URL 접속 가능) | ✅ |

### 된 것 — 권장 2개 (한계까지 명시)

- **서버사이드 필터·정렬.** 액션아이템 대시보드에서 담당자·상태로 거르고 기한으로 정렬한다.
    - 프론트에서 거르지 않고 **백엔드가 쿼리로 처리**한다 — `GET /api/action-items?assignee=&status=&sort_by=due_date`가 `query.filter()`·`order_by()`로 이어진다(`backend/main.py`).
    - *한계:* `due_date`가 자유 텍스트(String)라 "기한순 정렬"이 실제 날짜순이 아니라 **문자열 사전순**이다. 이 트레이드오프는 의도적으로 감수했다(이유·해결책은 3·6번).
- **완료체크 + 누적 대시보드.** 여러 회의의 액션아이템을 한 화면에 누적해 보고  담당자·상태 필터와 기한순 정렬을 지원한다. 각 항목이 소속 회의 제목 링크로 연결 - (`frontend/src/pages/DashboardPage.jsx`).
    - *한계:* 로그인이 없어 **모든 사용자가 전체 데이터를 공유**하며 , **조회·필터 전용 읽기 전용**이기에 실제 완료 체크(상태를 대기/진행/완료로 바꾸는 것)는 회의록 상세 화면(`frontend/src/pages/MeetingDetailPage.jsx`)에서 한다.

### 안 한 것 (의도적으로 미룸)

권장·보너스 중 나머지는 **필수 뼈대를 흔들지 않기 위해 후순위로 뒀다.** "다 하려다 뼈대를 못 세우는 것이 가장 나쁘다"는 과제 취지에 맞춘 판단이다.

- **로그인 / 사용자 권한 구분** — 간단한 인증과 사용자별 데이터 구분
    - 현재 누구나 모든 회의록·액션아이템에 접근 가능(6번에서 상술).
- UX 다듬기 — 입력 검증 / 에러 / 로딩 / 반응형.
- 회의록 파일 업로드 — 텍스트 파일 등 업로드 입력.

### 솔직하게 짚을 것 — LLM 모델

과제 요구는 "오픈라우터 **무료 티어** 모델"이고, 원래 무료 모델로 시작했다. 그런데 이 무료 모델이 **업스트림(Google AI Studio 등)에서 반복적으로 429(rate-limit)** 를 내서 분석이 자주 실패했다. 그래서 데모 안정성을 위해 **`.env`의 `OPENROUTER_MODEL` 값만 바꿔** 임시로 저가 유료 모델로 대체했다.

- 모델은 코드 수정 없이 **환경변수 한 줄로 교체**되게 짜여 있어(`openrouter_client.py`), 무료 모델로 되돌리는 건 값 변경 + 재시작뿐이다.
- 무료 모델의 429에 대비해 **재시도·선형 백오프(최대 3회)** 로직을 이미 넣어놨다(4번). 다만 업스트림 rate-limit는 애플리케이션 코드로 완전히 막을 수 없는 한계라, 리뷰 중 데모가 끊기지 않도록 유료 모델을 택했다.

## 2. 아키텍처

큰 그림은 **"프론트와 백엔드를 완전히 분리하고, 그 사이를 HTTP로 잇는다"** 이다. 프론트는 화면만, 백엔드는 REST API·DB·AI 호출만 담당한다.

### 전체 아키텍쳐

```
[브라우저]
   │  http://<EC2 퍼블릭 IP>  (포트 80)
   ▼
[Nginx]  ── 브라우저 요청에 의정적 파일(React 빌드 dist/) 서빙
   │
   ├─ "/api/..." 요청만 ──▶ [uvicorn:8000] (systemd 상시 실행)
   │                            │
   │                            ├─▶ [SQLite] 회의록·액션아이템 저장
   │                            └─▶ [OpenRouter] LLM 구조화 호출
   │
   └─ 그 외 요청 ──▶ React SPA(Single Page Application - html ) 
```

- **Nginx가 한 관문에서 두 갈래로 나눈다.** `/api`로 시작하면 백엔드(127.0.0.1:8000)로 넘기고, 나머지는 React 빌드 결과물을 보여준다.
    - 프론트·백엔드가 같은 주소(same-origin)를 쓰게 되어 배포 환경에선 CORS 문제가 사라진다.
- **uvicorn은 127.0.0.1로만 열어** 외부에 8000 포트를 직접 노출하지 않는다.(`127.0.0.1` 을 통해 같은 컴퓨터 안에서 오는 요청만 받음  즉 , 외부에서 직접 uvicorn에 접근 불가 ) ⇒ 오직 Nginx를 통해서만 접근된다.
    - 보안그룹도 22(SSH)·80(HTTP)만 열어 2차 보안
- **프론트 코드는 `/api` 상대경로만 안다.** 로컬에선 Vite proxy가, 배포에선 Nginx가 이 경로를 실제 백엔드로 연결한다. 즉 프론트 코드는 로컬·배포 구분 없이 하나로 유지되고, "누가 실제 백엔드로 연결하는지"만 환경별로 다르다.
- **Nginx 응답 대기 시간을 150초로 올렸다.** OpenRouter 호출은 재시도까지 겹치면 수십 초~2분이 걸릴 수 있는데, Nginx 기본값(60초)이면 백엔드가 응답하기 전에 연결을 끊는다. `proxy_read_timeout 150s`로 최외곽 타임아웃 계층을 맞췄다.
    - 타임아웃은 안쪽부터 바깥쪽으로 커야 한다(client 재시도 < 프론트 요청 timeout < Nginx).

### REST API (엔드포인트 10개)

프론트-백 분리를 실제 HTTP로 증명하는 지점. 모두 `/api` 프리픽스 아래 있다(`APIRouter(prefix="/api")`).

| 메서드 · 경로 | 하는 일 |
| --- | --- |
| `POST /api/analyze` | 회의록 텍스트 → LLM 구조화(JSON). DB는 안 건드림 |
| `POST /api/meetings` | 검토된 회의록  DB에 저장 (하위 액션아이템 함께) |
| `GET /api/meetings` | 회의록 목록 (최신순) 조회  |
| `GET /api/meetings/{id}` | 회의록 상세 (액션아이템 포함) |
| `PUT /api/meetings/{id}` | 회의록 수정 (제목·원문·결정·논의) |
| `DELETE /api/meetings/{id}` | 회의록 삭제 (액션아이템 cascade 삭제) |
| `POST /api/meetings/{id}/action-items`  | 특정 회의록에 새로운 액션아이템 추가  |
| `PUT /api/action-items/{id}` | 액션아이템 수정 (상태 포함) |
| `DELETE /api/action-items/{id}` | 액션아이템 삭제 |
| `GET /api/action-items` | 전체 액션아이템 (담당자·상태 필터, 기한 정렬) — 대시보드용 |

### 데이터 모델 (회의록 1 : 액션아이템 N)

```
meetings (1) ──< action_items (N)          [action_items.meeting_id → meetings.id]

meetings
  id(PK) · title · raw_text · decisions(JSON) · discussions(JSON)
  created_at · updated_at

action_items
  id(PK) · meeting_id(FK) · task · assignee(nullable) · due_date(nullable, String)
  status(대기/진행/완료, 기본 '대기') · created_at · updated_at
```

**액션아이템만 별도 테이블로 뺀 이유:** 액션아이템은 개별로 관리·수정·완료 체크의 대상이라 각자 고유 id(행)가 필요하다.

반면 결정사항·논의사항은 회의록에 종속된 단순 문자열 목록이라 생각하여  `meetings` 테이블의 JSON 컬럼에 담았다. 

( 한계점 : 결정사항·논의사항을 나중에 액션 아이템처럼 대시보드 형태로 발전 시키기 위해서는 따로 테이블을 분리해야 함 ) 

⇒ 이 차이가 수정 방식도 갈라놓는다 — 액션아이템은 id 기반 개별 API(PUT/DELETE), 결정·논의사항은 회의록 저장 시 배열 통째로 교체(PUT).

 회의록을 삭제하면 딸린 액션아이템도 함께 지워지도록 관계에 `cascade="all, delete-orphan"`을 걸었다.

## 3. 내가 내린 주요 결정과 그 이유

핵심 축은 **"각 선택을 내가 직접 설명할 수 있는가"** 였다. 편의성이 더 좋은 대안이 있어도, 자동화가 많아 내가 과정을 설명 못 하게 되는 도구는 이 과제 맥락에선 오히려 약점이라 판단했다.

**React + Vite (Next.js·CRA 대신)** — 프론트 개발이 처음이라, 초기 설계 단계의 판단 기준은 "막혔을 때 검색으로 풀릴 확률이 가장 높은 조합"이었다. React는 명세서 예시에도 포함돼 있고, 프론트 프레임워크 중 예제·튜토리얼·질문답변 자료가 압도적으로 많다. 

Vite는 지금 React 프로젝트를 시작하는 사실상의 표준 도구다(과거 표준이던 CRA는 유지보수가 중단됨). 즉 특별한 기능이 필요해서가 아니라, **처음 하는 개발·배포에서 표준을 따르면 자료가 많아 스스로 풀어나가기 쉽다**는 것이 핵심 판단이었다 — 이는 뒤에Nginx·Ubuntu·uvicorn을 고른 논리(업계 표준일수록 자료가 많다)와 같은 축이다.

**FastAPI + Python (백엔드)** — 백엔드는 초기 설계 때 내가 가장 익숙한 언어인 Python으로 정했다. 5일짜리 과제이고 커밋마다 "왜 이렇게 했는지"를 직접 설명해야 하는 평가 기준에서, 내가 가장 빠르게 이해할 수 있는 언어로 가는 게 맞다고 봤다. FastAPI는 그 Python의 현대 표준 웹 프레임워크이고, 요청/응답을 타입 기반으로 자동 검증하는 Pydantic을 내장하고 있다.  

**SQLite (개발·배포 공통)** — DB도 초기 설계 때 SQLite로 정했다. 별도 DB 서버를 띄우고 설정할 필요 없이 파일 하나로 관계형 모델(회의록 1:N 액션아이템)을 바로 세울 수 있어, "5일 안에 뼈대부터 관통"이라는 우선순위에 맞았다. 나중에 DB 연결 계층(`database.py`)을 만들 때, 연결 주소를 `DATABASE_URL` 환경변수로 빼서 PostgreSQL(RDS)로 바꿀 지점을 한 곳으로 좁혀뒀다 — 다만 실제 전환·검증은 하지 않았다(6번). 그래서 정확한 표현은 "이미 RDS를 쓸 수 있다"가 아니라 "교체 시 손댈 곳을 미리 한 줄로 좁혀뒀다"이다.

**fetch (axios 대신)** — 프론트의 API 호출 계층을 짜는 시점에 axios를 쓸지 검토했지만 브라우저 기본 내장 fetch로 갔다. axios의 대표 장점은 인터셉터(모든 요청·응답을 거치는 중간 처리 지점 — 예: 인증 토큰 자동 부착)인데 아키텍쳐 설계 당시 필수 6가지를 먼저 관통하자는 판단을 기반으로 로그인이 없어 그 이점이 없고, 엔드포인트가 10개 수준이라 fetch 반복 코드도 감당할 만했다. 인증을 붙이는 시점(로그인 구현 시)이 오면 axios를 다시 검토할 것이다.

**EC2 + Nginx (Lambda·Beanstalk·Caddy 대신)** — Lambda·Beanstalk는 플랫폼이 알아서 해주는 게 많아 "왜 이렇게 됐는가"를 내가 설명하기 어렵다. Caddy는 HTTPS를 자동 발급해 편하지만 그 과정을 설명 못 한다. EC2는 SSH로 한 줄씩 직접 제어하고, Nginx는 리버스 프록시를 직접 만져볼 수 있어 **투명성 = 설명 가능성**이 확보된다. 

**Ubuntu (Amazon Linux 대신)** —  EC2 인스턴스를 띄우는 단계에서 OS를 골라야 했다. "AWS 필수"는 EC2를 쓰라는 뜻이지 OS까지 강제하는 건 아니다. 배포가 처음이라, "FastAPI를 Ubuntu에 배포" 류의 자료가 가장 많은 Ubuntu를 택했다 — 막혔을 때 검색으로 풀릴 확률이 높은 쪽이다(Nginx를 고른 논리와 같은 축). 

**uvicorn 단일 (gunicorn 다중 워커 대신)** — uvicorn은 Day 1부터 FastAPI를 돌리는 기본 실행 서버로 써왔다(FastAPI가 표준으로 함께 쓰는 서버라, "골랐다"기보다 자연스럽게 따라온 것이다). 실제 판단은 배포 단계에서 나왔다 — "이대로 uvicorn 하나로 띄울까, gunicorn을 얹어 워커(요청을 처리하는 프로세스)를 여러 개로 늘릴까." 늘리지 않았다. 다중 워커는 요청이 몰릴 때 나눠 처리하려는 것인데, t3.micro는 CPU가 1개라 워커를 늘려도 효과가 작고 데모 트래픽엔 동시 요청 자체가 거의 없었다. 

### 순서를 이렇게 잡은 이유

**큰 순서는 백엔드 → 프론트 → 배포 → CSS였다.**

**백엔드를 먼저 세운 이유:** 의존 방향 때문이다. 백엔드는 화면 없이도 혼자 검증된다 — Day 1에 "텍스트→LLM→JSON"을 터미널 스크립트(`test_openrouter.py`)로 바로 돌려봤고, CRUD도 서버만 띄우면 확인된다. 반대로 프론트는 호출할 백엔드 API가 없으면 "분석하기"를 눌러도 테스트가 안 돼 반쪽짜리다. 그래서 독립적으로 완결되는 백엔드를 먼저 굳히고, 그 위에 프론트를 얹어 HTTP로 연결하는 순서가 자연스러웠다. 데이터가 실제로 오가는 뼈대를 먼저 세운 뒤 화면을 씌운 것이다.

백엔드 안에서도 **AI 구조화 → DB → REST API(CRUD)** 순으로 세웠는데, 이 순서 역시 의존 관계에서 나왔다. AI가 실제로 뱉는 출력 모양(결정사항, 논의사항..등)이 DB 스키마를 결정하기 때문에(어떤 필드를 어떤 타입으로 저장할지), 테이블을 설계하기 전에 "텍스트→LLM→JSON"부터 돌려 출력 형태를 먼저 고정했다. 그다음 그 형태를 담을 DB 모델을 세웠고, 마지막으로 DB에 데이터를 넣고 빼는 CRUD API를 그 위에 얹었다 — API는 조작할 테이블이 먼저 있어야 짤 수 있으니 DB 다음일 수밖에 없다. (단, DB를 건드리지 않는 `/analyze` 엔드포인트는 stateless 요청이라 AI 구조화와 함께 먼저 만들었다.)

**배포를 Day 4에 몰아둔 이유:** 가장 막힐 확률이 높은 구간이라 하루를 통째로 비워뒀다. Day 1~3에 "텍스트→LLM→저장→화면"을 로컬에서 완전히 관통시킨 뒤, 리스크가 큰 배포를 뒤에 몰아 시간을 확보하는 순서다.

(CSS를 맨 마지막에 둔 이유는 아래에 따로 적었다.)

### UI 스타일링을 마지막 순서로 잡은 이유

필수 기능·배포가 전부 끝난 뒤에야 스타일링을 시작했다. "동작하지만 못생긴 화면"으로 필수 관통을 먼저 증명하고, 남는 시간에만 다듬는 순서다 — 뼈대보다 겉모습을 먼저 신경 쓰다 관통을 못 하는 게 이 과제에서 가장 나쁜 결과라고 판단했다.

진행 방식도 커밋 단위 원칙을 그대로 따랐다. 공통 스타일(헤더·카드·폼 요소)을 먼저 한 커밋으로 정의하고, 그다음 화면(입력→검토·저장→목록→상세) 각각에 적용하는 커밋으로 나눴다 — 재료를 먼저 만들고 화면들이 재사용하는 구조라야 화면들이 같은 룩을 공유하고, 어느 커밋에서 뭐가 깨졌는지도 추적하기 쉽다. 이 작업 내내 **백엔드 스키마·API 응답 구조는 건드리지 않았다.** 마감이 임박한 시점에 배포된 백엔드까지 다시 손대는 건 리스크 대비 이득이 낮다고 봤다.

## 4. AI 구조화의 신뢰성 확보

**"AI가 지어낸 제목·담당자·기한이 그대로 저장되는 일"을 막는 것이 핵심이라고 생각하였고, 이를 다층 방어로 처리했다.** AI 출력은 신뢰하지 않는다는 전제에서 출발한다.

**(1) 프롬프트 — 형식·내용 강제 (1차 방어)**

- 시스템 프롬프트에 JSON 스키마를 명시하고, 설명 없이 JSON만 출력하도록 지시.
    - 프로그램끼리 데이터를 주고받는 사실상의 표준 형식이기 때문이다.
- "원문에 담당자·기한이 없으면 반드시 `null`, 절대 지어내지 마라"는 규칙을 명시.
- 추출할 내용이 없으면 모든 배열을 빈 배열로 반환하도록 지시.
- `temperature: 0` + `response_format: json_object`로 출력 흔들림을 최소화.

**(2) 후처리 — 구조 방어 (2차 방어)**

- 일부 모델이 ````json` 코드블록으로 감싸주는 경우를 대비해 벗겨낸 뒤 파싱.
- 모델이 특정 키를 통째로 빼고 줄 경우를 대비해 `setdefault`로 누락 키를 빈 배열로 보정.

**(3) DB 설계 — 저장 방어**

- `assignee`·`due_date`를 **nullable**로 설계. 원문에 없으면 비워둔 채 저장할 수 있게 하는 것 자체가 hallucination 방어와 직결된다.

**(4) 사용자 검토 — 최종 방어 (필수 4번과 연결)**

- AI 결과를 그대로 신뢰해 저장하지 않는다. 검토 화면(`ReviewPage`)에서 제목·결정사항·논의사항·액션아이템(할일·담당자·기한)을 전부 수정한 뒤 저장한다. 담당자·기한처럼 판단이 필요한 값의 마지막 방어선은 사람이다.

**(+) 호출 자체의 신뢰성 — 429 재시도**

무료 모델이 자주 429(rate-limit)를 내서, 신뢰성 로직에 **재시도 + 선형 백오프(최대 3회, 요청 timeout 30초)** 를 넣었다. 이건 "AI가 틀린 값을 준다"와는 다른 축(호출 자체가 실패)이지만, AI 결과를 **안정적으로 받아내는 것**도 신뢰성의 일부라 함께 처리했다.

## 5. 막혔던 지점과 어떻게 풀었는지

막힌 지점은 크게 두 구간이었다 — **로컬 개발 단계(AI 호출의 타임아웃·안정성, 프론트 에러 처리)**와 **AWS 배포 단계(네트워크·권한·설정)**. 특히 배포에서는 서로 다른 계층에서 순서대로 막혔고, 계층을 좁혀가며 원인을 분리한 과정이 핵심이다.

#### 로컬 개발에서 막힌 것

**(A) 타임아웃 계층 역전 — 요청이 자꾸 끊김**

Day 1 테스트에서 AI 분석 요청이 자꾸 중간에 끊겼다. 원인을 파보니, AI 호출 내부의 재시도(백오프 5→10→15초)가 30초를 넘길 수 있는데 그 요청을 감싸는 바깥쪽 타임아웃이 그보다 짧아서, 안쪽이 아직 재시도 중인데 바깥이 먼저 끊는 상황이었다. → 타임아웃은 **안쪽이 짧고 바깥쪽이 길어야 한다**는 걸 정리하고, 요청측을 90초로 늘려 해결했다. 이 원칙은 나중에 배포 단계에서 Nginx 최외곽 타임아웃을 150초로 잡는 판단으로 그대로 이어졌다.

**(B) 무료 모델 429 — 분석이 자주 실패**

무료 LLM이 업스트림(Google AI Studio 등) 용량 부족으로 429(rate-limit)를 자주 반환해 분석이 실패했다. → 재시도 + 선형 백오프(최대 3회)로 완화했다. 다만 이건 애플리케이션 코드로 **완전히는 못 막는 구조적 한계**다(호출량이 전 세계 공유풀에 달려 있음). 그래서 데모 안정성을 위해 임시로 저가 유료 모델로 대체했고, 이 판단은 1번에 솔직히 적었다.

**(C) 타임아웃 요청이 별도 에러 경로로 빠짐 — 영어 메시지 노출**

프론트 API 계층에서, `!res.ok`로는 응답이 온 실패(4xx/5xx)만 잡히고 **타임아웃으로 강제 종료된 요청은 `AbortError`라는 다른 에러**로 빠져 브라우저 기본 영어 메시지가 그대로 사용자에게 노출됐다. 입력 화면(`InputPage`) 작업 중 실제로 이 경로를 타보면서 발견했다. → 공통 요청 함수(`request()`)에서 `error.name === 'AbortError'`를 따로 잡아 한국어 메시지로 변환해 해결했다. 필수 3번(비동기·타임아웃 처리)을 "성공했다 치고" 넘기지 않고 실패 경로까지 실제로 확인한 지점이다.

**(D) 액션아이템 가변 개수 vs Rules of Hooks 충돌**

액션아이템은 사용자가 추가·삭제해서 개수가 계속 변한다. 그런데 React에는 "상태(`useState`)를 매번 같은 개수로 호출해야 한다"는 규칙(Rules of Hooks)이 있어서, 부모 화면 안에 항목 수만큼 상태를 만들려 하면 개수가 변할 때 규칙을 어긴다. → 액션아이템 한 줄을 `ActionItemRow`라는 별도 컴포넌트로 떼어냈다. 부모는 이 컴포넌트를 항목 개수만큼 반복해서 그리기만 하고, 각 줄이 자기 상태를 스스로 관리한다. 컴포넌트 하나는 항상 같은 개수의 상태를 부르니 규칙을 지키고, 그걸 여러 번 그리는 건 자유라 가변 개수가 해결된다.

#### AWS 배포에서 막힌 것

**(0) 빌드 실패 — 도구 버전 불일치**

EC2 기본 Node 18과 Vite가 요구하는 버전(20+)이 안 맞아 `npm run build`가 `node:util`의 export를 못 찾고 SyntaxError. → NodeSource로 Node 20 재설치, `node_modules` 삭제 후 재설치로 해결.

**(1) 네트워크 계층 — 접속 자체가 안 됨**

브라우저에서 `ERR_CONNECTION_TIMED_OUT`. → AWS 보안그룹 인바운드에 SSH(22)만 있고 HTTP(80)가 실제로는 없었다. 80/Anywhere를 추가해 해결.

**(2) OS 파일 권한 계층 — Nginx 500**          

Nginx가 500, 에러 로그에 `stat() dist/index.html failed (13: Permission denied)`. → `dist` 폴더와 `index.html` 자체 권한은 정상이었다. 원인은 상위 경로인 `/home/ubuntu` 홈 디렉토리 기본 권한(750)이었다. 리눅스는 목적지 파일뿐 아니라 경로상의 모든 디렉토리에 통과권(`x`)이 있어야 접근되는데, Nginx 실행 주체인 `www-data`(other)가 홈 앞에서 막힌 것. `chmod o+x /home/ubuntu`로 통과권(`x`)만 부여해 해결. 
> 배포 IP는 Elastic IP를 붙이지 않아 인스턴스 재부팅 시 바뀔 수 있다. 보안그룹은 SSH(22)는 내 IP로 제한하고 앱 포트(80)만 공개했다. 데모 기간엔 접속 IP를 최종 확인해야 하고, 운영이라면 고정 IP(Elastic IP) 환경으로 가는 게 맞다.

## 6. AI(바이브코딩)를 어디에 어떻게 썼는지 + 시간이 더 있었다면

### AI 활용

- **Claude Code (로컬):** 실제 코드 생성·실행·디버깅·커밋. 단, **커밋마다 "왜 이렇게 했는가"를 스스로 설명할 수 있는 상태를 넘어가는 기준으로 삼았다.** 주석은 직접 달았다.
- **설계·개념 판단:** 아키텍처 선택, 에러 처리 계층, DB 모델링 등 "왜 이 구조인가"를 검증하는 데 사용.
- **개념 학습 보조:** FastAPI 요청 흐름, 비동기의 두 층위, 504/502 구분 등 낯선 개념을 잡는 데 사용(이해용).
- **UI 스타일링(CSS/레이아웃):** 스크린샷을 확인하며 AI와 반복 조정. 색상 hex·여백·버튼 스타일 같은 세부 값은 AI 제안을 보고 마음에 안 들면 다른 방향을 요청하는 식으로 결정했다. 다만 "무엇을 지킬지"(API 호출·상태관리 로직, 백엔드 스키마는 절대 변경하지 않는다)라는 범위 판단은 직접 세우고 매 지시마다 지켜졌는지 확인했다.

### 시간이 더 있었다면

- **인증·인가:** 현재는 누구나 모든 회의록에 접근 가능하다. User 테이블 + role 필드 + `meeting_id` 소유권 기반 접근 제어를 넣었을 것이다. *(CORS는 "어느 사이트가 부를 수 있는가"이고, 인증·인가는 "누가 무엇을 할 수 있는가"로 별개 문제다.)*
- **무료 모델 재전환:** `OPENROUTER_MODEL`을 무료 모델로 되돌리고, 업스트림 429를 견디도록 백오프를 늘리거나 여러 무료 모델을 폴백 체인으로 묶는 방법을 검토했을 것이다.
- **정규화:** 결정·논의사항을 별도 테이블로 분리, 실제 회의 일자 필드 추가(현재는 `created_at`만).
- **`due_date` 타입 전환:** `due_date`를 String으로 둔 이유는 hallucination 방지(AI가 애매한 날짜 표현을 억지로 정확한 날짜로 지어내지 않게)와 사용자 수정 자유도("다음 주까지" 같은 자연스러운 표현 허용) 둘 다였다. 다만 이 설계는 대시보드의 "기한순 정렬"과 충돌한다 — 자유 텍스트라 실제 날짜 임박 순이 아닌 문자열 사전순으로만 정렬된다. 실제 날짜 정렬이 필요해지면 Date 타입 전환 + AI 프롬프트 형식 강제 + 프론트 date picker 교체 + 기존 데이터 마이그레이션까지 필요하다.
- **PUT → PATCH:** 부분 수정 지원.
- **운영 다듬기:** 실서비스라면 HTTPError 원문을 사용자에게 그대로 노출하지 않고 일반 메시지 + 서버 로그로 분리, 트래픽이 늘면 gunicorn 다중 워커 검토.
- **RDS(PostgreSQL) 실제 전환** — 배포는 SQLite로 완주. `database.py`에서 `DATABASE_URL`로 교체할 **지점 한 줄**은 만들어뒀으나, 실제 RDS 전환·검증은 안 했다(3·6번).
- UI / UX 다듬기 — 입력 검증 / 에러 / 로딩 / 반응형 등 다듬기
- 회의록 파일 업로드 — 텍스트 파일 등 업로드 입력 구현
